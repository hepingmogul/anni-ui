## 一、背景与目标

当前项目（`@anni-ui/react`）共有 18 个组件，**暂无任何自动化测试**，逻辑正确性依赖人工验收。随着组件数量增加和后续迭代（Select、Modal、Table 等），人工验收成本急剧上升，引入自动化单元测试已是刚需。

**本阶段目标：**

- 搭建完整的 **Vitest + @testing-library/react** 测试环境
- 为现有全部 18 个组件编写核心测试用例
- 建立测试目录规范和测试编写约定，供后续组件开发参考
- 目标语句覆盖率 **≥ 80%**

---

## 二、技术选型

| 技术 | 选择 | 说明 |
|------|------|------|
| 测试框架 | **Vitest** | 与 Vite 同生态，配置共用，ESM 原生支持，速度极快；无需额外转译配置 |
| DOM 模拟 | **jsdom** | Vitest 内置支持，通过 `environment: 'jsdom'` 启用；足够覆盖所有 UI 组件测试场景 |
| React 测试工具 | **@testing-library/react** | 以用户视角操作组件（查询、点击、输入），避免测试实现细节 |
| 用户事件模拟 | **@testing-library/user-event** | 比 `fireEvent` 更贴近真实用户行为（键盘事件序列、焦点管理等） |
| 扩展匹配器 | **@testing-library/jest-dom** | 提供 `toBeInTheDocument()`、`toHaveClass()`、`toBeDisabled()` 等语义化断言 |
| 覆盖率工具 | **@vitest/coverage-v8** | V8 原生覆盖率，性能优于 istanbul，Vitest 官方推荐 |

**不选 Jest 的原因：** 项目已使用 Vite + ESM，Jest 需要额外的 `babel-jest` 或 `ts-jest` 转译配置；Vitest 与 Vite 共享同一份 `vite.config.ts`，零额外配置成本。

## 四、配置方案

### 4.1 vite.config.ts 改造

在现有 `vite.config.ts` 中追加 `test` 配置块（Vitest 支持直接复用 Vite 配置文件）：

```ts
// packages/react/vite.config.ts
import { defineConfig } from 'vite'
import react from '@vitejs/plugin-react'

export default defineConfig({
  plugins: [react()],
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/test/setup.ts'],
    coverage: {
      provider: 'v8',
      reporter: ['text', 'html', 'lcov'],
      include: ['src/core/**/*.{ts,tsx}'],
      exclude: [
        'src/core/**/index.ts',
        'src/core/**/*.types.ts',
      ],
      thresholds: {
        statements: 80,
        branches: 80,
        functions: 80,
        lines: 80,
      },
    },
  },
})
```

> **说明：**
> - `globals: true`：测试文件中无需手动 `import { describe, it, expect } from 'vitest'`，与 Jest 写法完全兼容
> - `setupFiles`：每个测试文件执行前自动运行，用于引入 jest-dom 扩展匹配器
> - `include` 仅覆盖 `src/core/`，排除纯类型文件和桶导出文件，避免虚假覆盖数字

### 4.2 vitest.config.ts（可选独立配置）

如果后期需要将测试配置与构建配置严格分离（例如 CI 环境只跑测试），可创建独立的 `vitest.config.ts`：

```ts
// packages/react/vitest.config.ts（可选，与 vite.config.ts 方案二选一）
import { defineConfig, mergeConfig } from 'vitest/config'
import viteConfig from './vite.config'

export default mergeConfig(viteConfig, defineConfig({
  test: {
    environment: 'jsdom',
    globals: true,
    setupFiles: ['./src/test/setup.ts'],
  },
}))
```

### 4.3 tsconfig.json 调整

确保 TypeScript 能识别 `jest-dom` 的类型扩展和 Vitest 全局变量：

```jsonc
// packages/react/tsconfig.json（在 compilerOptions.types 中追加）
{
  "compilerOptions": {
    "types": ["vitest/globals", "@testing-library/jest-dom"]
  },
  "include": ["src"]
}
```

### 4.4 setup 文件

创建测试全局 setup 文件，引入 jest-dom 扩展匹配器：

```ts
// packages/react/src/test/setup.ts
import '@testing-library/jest-dom'
```

> 该文件由 `vite.config.ts` 中的 `setupFiles` 自动引入，无需在每个测试文件中手动 import。

### 4.5 package.json 脚本

在 `packages/react/package.json` 的 `scripts` 中追加：

```jsonc
{
  "scripts": {
    "dev": "vite",
    "build": "tsc -b && vite build",
    "preview": "vite preview",
    "test": "vitest",
    "test:run": "vitest run",
    "test:coverage": "vitest run --coverage"
  }
}
```

| 命令 | 说明 |
|------|------|
| `pnpm test` | 监听模式，文件变更时自动重跑（开发时使用） |
| `pnpm test:run` | 单次运行，不监听（CI 使用） |
| `pnpm test:coverage` | 单次运行并生成覆盖率报告 |

## 五、测试覆盖范围

### 5.1 纳入测试的内容

| 类型 | 说明 |
|------|------|
| 组件渲染 | 默认 Props 下组件正确挂载、DOM 结构符合预期 |
| Props 生效 | 各 Props 变化时 UI 表现正确（类名、属性、文本等） |
| 用户交互 | 点击、输入、键盘操作触发正确的回调和状态变化 |
| 受控 / 非受控 | 受控模式下外部 value 驱动 UI；非受控模式下内部状态自维护 |
| 无障碍属性 | aria 属性正确挂载（aria-invalid、aria-label、role 等） |
| 边界条件 | disabled 状态、超出范围值、空值等异常输入的处理 |

### 5.2 暂不纳入测试的内容

| 类型 | 原因 |
|------|------|
| Tailwind 类名精确匹配 | 类名受构建影响，容易误报；用 `toHaveClass` 只断言关键类名 |
| CSS 视觉呈现（颜色、尺寸像素值） | jsdom 无法计算 CSS，应由视觉测试工具（如 Storybook + Chromatic）负责 |
| 动画过渡效果 | jsdom 不执行 CSS transition/animation |
| 端对端流程 | 留给后续 Playwright/Cypress 阶段 |

## 六、测试分层策略

```
单元测试（本阶段目标）
├── 组件渲染测试          - 验证 DOM 结构和属性
├── 交互测试              - 验证用户操作后的状态变化
├── Props 契约测试        - 验证各 Prop 按文档描述生效
└── 无障碍属性测试        - 验证 aria 属性正确挂载

（后续阶段）
├── 集成测试              - 验证组件组合场景（如 Form + Input）
└── 视觉回归测试          - Storybook + Chromatic
```

**单个测试文件内的结构约定：**

```ts
describe('ComponentName', () => {
  describe('渲染', () => { /* 默认渲染、Props 渲染 */ })
  describe('交互', () => { /* 点击、输入等事件 */ })
  describe('无障碍', () => { /* aria 属性验证 */ })
  describe('边界条件', () => { /* disabled、空值等 */ })
})
```

```
src/core/
├── Button/
│   ├── index.ts
│   ├── Button.tsx
│   ├── Button.types.ts
│   └── Button.test.tsx        ← 测试文件
├── Input/
│   ├── index.ts
│   ├── Input.tsx
│   ├── Input.types.ts
│   └── Input.test.tsx
├── Form/
│   ├── index.ts
│   ├── Form.tsx
│   ├── Form.types.ts
│   └── Form.test.tsx          ← 表单体系集成测试
└── Theme/
    ├── ThemeProvider.tsx
    └── ThemeProvider.test.tsx
```

全局测试工具统一放置在：

```
src/test/
├── setup.ts                   ← jest-dom 引入（被 setupFiles 自动加载）
└── utils.tsx                  ← 公共 render 包装（如注入 ThemeProvider）
```

`src/test/utils.tsx` 示例（封装带 Provider 的 render）：

```tsx
// src/test/utils.tsx
import { render, type RenderOptions } from '@testing-library/react'
import { ThemeProvider } from '../core/Theme/ThemeProvider'

const AllProviders = ({ children }: { children: React.ReactNode }) => (
  <ThemeProvider defaultMode="light">{children}</ThemeProvider>
)

const customRender = (ui: React.ReactElement, options?: RenderOptions) =>
  render(ui, { wrapper: AllProviders, ...options })

export * from '@testing-library/react'
export { customRender as render }
```

> **约定：** 所有测试文件统一从 `../../test/utils` 导入 `render`，而非直接从 `@testing-library/react`，确保每次渲染都在主题上下文中运行。

## 九、测试代码示例

### 9.1 Button 测试（基础交互）

```tsx
// src/core/Button/Button.test.tsx
import { describe, it, expect, vi } from 'vitest'
import { render, screen } from '../../test/utils'
```

```tsx
    vi.useRealTimers()
  })

    expect(screen.getByText('操作成功')).toBeInTheDocument()
  })

    act(() => { Toast.success('临时消息', { duration: 3000 }) })
    expect(screen.getByText('临时消息')).toBeInTheDocument()
    act(() => { vi.advanceTimersByTime(3000) })
    expect(screen.queryByText('临时消息')).not.toBeInTheDocument()
  })
})
```

```tsx
// src/core/Theme/ThemeProvider.test.tsx
import { describe, it, expect } from 'vitest'
import { render, screen, act } from '@testing-library/react'
import userEvent from '@testing-library/user-event'
import { ThemeProvider } from './ThemeProvider'
import { useTheme } from './useTheme'

const ThemeConsumer = () => {
  const { mode, setMode, toggle } = useTheme()
  return (
    <div>
      <span data-testid="mode">{mode}</span>
      <button onClick={() => setMode('dark')}>切换暗色</button>
      <button onClick={toggle}>切换</button>
    </div>
  )
}

describe('ThemeProvider', () => {
  it('默认 light 模式下 html 无 dark 类', () => {
    render(<ThemeProvider defaultMode="light"><ThemeConsumer /></ThemeProvider>)
    expect(document.documentElement).not.toHaveClass('dark')
  })

  it('切换到 dark 后 html 添加 dark 类', async () => {
    render(<ThemeProvider defaultMode="light"><ThemeConsumer /></ThemeProvider>)
    await userEvent.click(screen.getByText('切换暗色'))
    expect(document.documentElement).toHaveClass('dark')
  })
})
```

|------|------|
